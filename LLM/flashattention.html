背景：为什么 Attention 这么慢？

1.1 标准 Self-Attention 的计算与内存复杂度

标准的 Scaled Dot-Product Attention 定义为：

$$\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$

其中 \(Q, K, V \in \mathbb{R}^{N \times d}\)，\(N\) 为序列长度，\(d\) 为头维度。核心问题在于 \(N \times N\) 的中间注意力矩阵：

操作计算复杂度内存复杂度
QKᵀ 矩阵乘法\(O(N^2 d)\)\(O(N^2)\)
Softmax\(O(N^2)\)\(O(N^2)\)
Attention × V\(O(N^2 d)\)\(O(Nd)\)
合计\(O(N^2 d)\)\(O(N^2)\)
⚠️
当 N = 16384（16k context），float16 下注意力矩阵占用约 512 MB，完全超过 GPU 的 L2 Cache，必须频繁读写 HBM——这才是真正的瓶颈所在。

1.2 GPU 内存层次结构：IO 才是瓶颈

理解 FlashAttention 的核心前提是了解 GPU 的存储层次。带宽越高越靠近 SM，但容量也越小：

Registers
~几KB
~数十 TB/s
L1 / Shared Memory
~几百KB
~10 TB/s
L2 Cache
~几十MB
~4 TB/s
HBM (Global Memory)
几十~几百 GB
~2-3 TB/s

根本问题：标准 Attention 频繁将 \(N \times N\) 的中间矩阵写入 HBM 再读回，HBM I/O 复杂度高达 \(\Theta(N^2 d)\)。FlashAttention 的核心目标：让计算完全在 SRAM 内完成，大幅减少 HBM 访问次数。

2.1 核心思想：Tiling + Online Softmax

FlashAttention v1 的核心是分块（Tiling）与在线 Softmax（Online Softmax）的结合：将 Attention 计算分解为多个小块，使每块完全在 SRAM 中完成，避免将中间结果写回 HBM。

2.2 在线 Softmax 算法

经典 Softmax 需要两趟扫描：第一趟求最大值，第二趟计算指数和归一化。FlashAttention v1 利用数值稳定的在线算法，将两趟合并为一趟，维护三个状态量：

// 维护状态：当前已处理前 i 列

m
\(m_i = \max(s_1, \ldots, s_i)\)  —— 局部最大值（防止上溢）
ℓ
\(\ell_i = \sum_{j=1}^{i} e^{s_j - m_i}\)  —— 局部归一化因子
O
\(O_i\)  —— 当前输出累积

新增第 \(i+1\) 列时的更新公式：

$$m_{i+1} = \max(m_i,\ s_{i+1})$$ $$\ell_{i+1} = e^{m_i - m_{i+1}} \cdot \ell_i + e^{s_{i+1} - m_{i+1}}$$ $$O_{i+1} = \frac{e^{m_i - m_{i+1}} \cdot \ell_i \cdot O_i + e^{s_{i+1} - m_{i+1}} \cdot v_{i+1}}{\ell_{i+1}}$$

2.3 分块算法流程

Input:  Q, K, V ∈ ℝ^{N×d}，块大小 B_r, B_c
Output: O ∈ ℝ^{N×d}

将 Q 分成 T_r = ⌈N/B_r⌉ 块；将 K, V 分成 T_c = ⌈N/B_c⌉ 块

for i = 1 to T_r:                    # 外层：遍历 Q 块
    从 HBM 加载 Q_i → SRAM
    初始化 O_i = 0, ℓ_i = 0, m_i = -∞
    
    for j = 1 to T_c:                # 内层：遍历 K, V 块
        从 HBM 加载 K_j, V_j → SRAM
        
        # ✅ 以下所有计算均在 SRAM 内完成
        S_ij = Q_i × K_j^T / √d      # (B_r × B_c)
        m̃_ij = rowmax(S_ij)
        P̃_ij = exp(S_ij - m̃_ij)
        ℓ̃_ij = rowsum(P̃_ij)
        
        # 在线状态更新
        m_new = max(m_i, m̃_ij)
        ℓ_new = exp(m_i-m_new)·ℓ_i + exp(m̃_ij-m_new)·ℓ̃_ij
        O_i  ← (ℓ_i·exp(m_i-m_new)·O_i + exp(m̃_ij-m_new)·P̃_ij·V_j) / ℓ_new
        m_i ← m_new; ℓ_i ← ℓ_new
    
    将 O_i 写回 HBM  # 仅写一次！

3.1 v1 的瓶颈

v1 主要优化了 HBM I/O，但在 GPU 算力利用率上仍有三大不足：

  • 非矩阵乘法操作过多：每轮迭代的 rescaling 操作（乘除法）占比过高，而 GPU Tensor Core 只对 GEMM（矩阵乘法）高效
  • 并行化维度不足：序列维度上无法并行，无法充分利用 SM 数量
  • Warp 间负载不均衡：work partitioning 策略导致 warp 同步开销

3.2 三大核心改进

// 改进 01
减少非 GEMM 操作
将 rescaling 推迟到循环结束，维护未归一化输出 \(\tilde{O}\)，最后一次性除以 \(\ell\)。非矩阵乘法操作数减少约 2×。
// 改进 02
Q 外循环策略
将 Q 放在外循环，不同 Q 块间完全独立，序列维度可在前向和反向 pass 中完全并行，充分利用多 SM。
// 改进 03
Split-Q Warp 策略
从 split-K 改为 split-Q，不同 warp 处理不同 Q 行，消除 warp 间通信，降低同步开销。

3.3 改进1 详解：推迟 Rescaling

// v2 循环中维护未归一化输出

$$\tilde{O}^{(j)} = e^{m^{(j-1)} - m^{(j)}} \tilde{O}^{(j-1)} + e^{S_j - m^{(j)}} V_j$$

// 循环结束后一次归一化

$$O = \tilde{O}^{(T_c)} \;/\; \ell^{(T_c)}$$

3.4 改进3 详解：Warp 分工对比

❌ v1 / Split-K（需要 warp 间同步）
Warp 0: Q[0:16] × K[ 0: 32]
Warp 1: Q[0:16] × K[32: 64]  # 需要 reduce ↓
Warp 2: Q[0:16] × K[64: 96]  # 需要 reduce ↓
Warp 3: Q[0:16] × K[96:128]  # 需要 reduce ↓
        → warp 间通信 + 共享内存同步
✅ v2 / Split-Q（完全独立）
Warp 0: Q[ 0:16] × K[0:128]  # 独立 ✓
Warp 1: Q[16:32] × K[0:128]  # 独立 ✓
Warp 2: Q[32:48] × K[0:128]  # 独立 ✓
Warp 3: Q[48:64] × K[0:128]  # 独立 ✓
        → 零同步开销

4.1 新硬件特性：Hopper GPU (H100)

FA v3 专门针对 NVIDIA Hopper 架构（H100/H200）进行深度优化，充分利用三个关键新硬件特性：

⚡
TMA
Tensor Memory Accelerator
专用异步 DMA 引擎，可在计算进行时异步传输数据，无需占用 SM 资源。
🔥
WGMMA
Warpgroup Matrix Multiply
新一代矩阵乘法指令，吞吐量是 Ampere HMMA 的 2×，延迟也更低。
🎯
FP8 Tensor Core
支持 E4M3/E5M2 格式，计算吞吐量是 FP16 的 2×，显著加速推理场景。

4.2 生产者-消费者异步流水线

FA v3 的核心创新：将 warp 分为生产者（Producer）和消费者（Consumer）两组，利用 TMA 实现数据加载与计算的完全重叠：

// 软件流水线示意（时间从左到右）

无流水线：

Load K₁V₁
Compute₁
Load K₂V₂
Compute₂
Load K₃V₃
Compute₃

有流水线（加载延迟被完全隐藏）：

Load K₁V₁
Load K₂V₂
Compute₁
Load K₃V₃
Compute₂
Compute₃

4.3 两阶段 Attention：WGMMA 与 Softmax 重叠

为了进一步将矩阵乘法与 Softmax 计算重叠，FA v3 采用两阶段算法，利用 WGMMA 的异步执行窗口：

①

阶段1：Score 矩阵计算

\(S_j = Q_i K_j^T / \sqrt{d}\)　（启动 WGMMA，异步执行）

②

阶段2：同时执行

WGMMA（异步）

\(\tilde{O}_j \mathrel{+}= \tilde{P}_j \cdot V_j\)

Softmax（同时在CPU执行）

\(m = \text{rowmax}(S_j)\)
\(\tilde{P}_j = e^{S_j - m}\)
\(\ell = \text{rowsum}(\tilde{P}_j)\)

4.4 FP8 低精度支持

🔬
惰性非相干处理（Incoherent Processing）：对 \(Q\) 和 \(K\) 预乘以随机哈达玛矩阵 \(H\)（正交矩阵），将极端值"分散"开，使 FP8 表示更稳定： $$Q' = QH,\quad K' = KH$$ 由于 \(H\) 正交，\(Q'K'^T = QHH^TK^T = QK^T\)，不改变最终 Attention 值，同时大幅降低 FP8 量化误差。

数学推导：分块计算的正确性

6.1 数值稳定 Softmax

标准 Softmax 在 \(x_i\) 较大时会上溢，数值稳定版本减去全局最大值 \(m = \max_j x_j\)：

$$\text{softmax}(x)_i = \frac{e^{x_i - m}}{\sum_{j=1}^N e^{x_j - m}},\quad m = \max_j x_j$$

6.2 分块 Softmax 的合并证明

设 \(x = [x^{(1)}, x^{(2)}]\) 被分为两块，分别计算各块的局部统计量，然后合并：

1
各块统计量：\(\quad m^{(k)} = \max x^{(k)},\quad \ell^{(k)} = \sum e^{x^{(k)} - m^{(k)}}\)
2
全局最大值：\(\quad m = \max(m^{(1)}, m^{(2)})\)
3
合并归一化因子：\(\quad \ell = e^{m^{(1)}-m} \ell^{(1)} + e^{m^{(2)}-m} \ell^{(2)}\)
✓
则 \(\text{softmax}(x)^{(k)}_i = \frac{e^{x^{(k)}_i - m^{(k)}} \cdot e^{m^{(k)} - m}}{\ell}\) — 与全局计算结果完全一致

6.3 分块 Attention 输出的合并

设注意力得分被分为两块 \(S^{(1)}, S^{(2)}\)，对应值矩阵 \(V_1, V_2\)，最终输出等于：

$$O = \text{diag}(\ell)^{-1}\!\left(e^{m^{(1)}-m}\text{diag}(\ell^{(1)})\tilde{O}^{(1)} + e^{m^{(2)}-m}\text{diag}(\ell^{(2)})\tilde{O}^{(2)}\right)$$ $$= \text{softmax}\!\left([S^{(1)}, S^{(2)}]\right)\begin{bmatrix}V_1\\V_2\end{bmatrix} \quad \checkmark$$

这证明了：在 SRAM 中对小块独立计算后合并，与对完整矩阵一次性计算，结果完全数学等价。

工程实现要点

7.1 块大小的选择

块大小 \(B_r, B_c\) 需满足 SRAM 容量约束（同时存放 Q、K、V 子块和中间结果）：

$$B_r \cdot d + B_c \cdot d + B_r \cdot B_c \leq M_{\text{SRAM}}$$

典型取值（d=64，A100 SRAM ~100KB per SM）：\(B_r = B_c = 64\) 或 128，需根据 \(d\) 和硬件参数调整。

7.2 因果掩码的高效处理

⚡
对于 Decoder 的因果 Attention（下三角掩码），FlashAttention 只对对角线附近的块应用掩码，远离对角线的下三角块和完整上三角块无需特殊处理——这使因果 Attention 的计算量自然减少约 50%（三角矩阵 vs 全矩阵）。

7.3 反向传播：重计算策略

阶段存储内容不存储
前向 pass\(m, \ell\)（各 \(O(N)\)），输出 \(O\)\(P = \text{softmax}(QK^T/\sqrt{d})\)（\(O(N^2)\)）
反向 pass利用 \(Q,K,V,O,m,\ell\) 重新计算 \(P\)无需保留注意力矩阵

额外计算 FLOPs 约为前向的 1.1–1.2×，但节省了 \(O(N^2)\) 的 HBM 写入和读取。

7.4 多头注意力的并行化

不同的 head 和 batch 维度完全独立，各自映射到不同的 SM 上并行执行，充分利用 GPU 的大规模并行性——这也是为什么 FlashAttention 在头数和批大小较大时效益最显著。
